The short version

Bottom line

The tool can now read product packaging photos and pull out facts the feed text doesn't contain — exact dosages, probiotic strain codes, ingredient forms. We tested this on 5 Metagenics products with a controlled comparison (same prompts, same model, with vs. without the image). The image-driven output is genuinely richer, and we can prove the model isn't just making it up.

This delivers the second of the three action items from the 29 April catch-up:

  • Done   Build web fetch into the tool — scrape product detail pages, including image extraction.
  • Next   Test on RJ Living (furniture) and Bydee (apparel) — different industries will stress-test different attributes (dimensions, materials, colour, sizes).
  • Planned   Wire in a pre-onboarding audit step (the kind of audit work James has been running) so structural feed issues get caught before enhancement begins.

Why this matters

Most product feeds we see in the wild have minimal description text. Two of the five Metagenics products we tested have descriptions under 50 words — no ingredient list, no dosages, no strain names. With nothing else to work with, an AI rewrite can only reorganise what's already there, or guess.

Product packaging photos, on the other hand, almost always contain the structured information shoppers actually search for: pack size, dosage form, active ingredient names, strain codes, allergen flags. Feeding those photos to the model alongside the text closes the gap — without us having to ask the merchant to fix their data first.

How it works, end to end

The merchant gives us a feed URL. From there, the tool runs the following steps, in order. Steps 3 and 4 are the new pieces we built since the 29 April catch-up.

1

Read the merchant's feed

What
Download the feed file (XML, CSV, or TSV) and turn each product into a normalised record.
Why
Every merchant exports their feed differently. We standardise it first so the rest of the pipeline doesn't have to care about format quirks.
This run 296 Metagenics products parsed from a DataFeedWatch XML feed.
2

Validate against Google's spec

What
Check each product against Google Merchant Center's rules — required fields, formatting, allowed values, country-specific extras.
Why
We need to know what's already correct and what needs fixing or filling in. This also flags issues the merchant must fix at the source (GTINs, item group IDs).
This run Flagged missing google_product_category and a couple of availability formatting fixes — both auto-fillable by the tool.
3

Scrape each product's web page NEW

What
For each product, fetch its product detail page on the merchant's website. The scraper tries the structured data Google uses (JSON-LD) first, then falls back to standard HTML tags, then a configurable selector for tricky pages.
Why
The feed alone often doesn't contain enough context. The PDP usually does — and it's always the source of truth.
This run 4 of 5 PDPs returned structured data successfully. The 5th was a DEV record whose page was effectively empty — the pipeline handled this gracefully and continued with text-only for that one.
4

Extract packaging image URLs NEW

What
Pull the URLs of the product's packaging photos from the PDP we just scraped. Capped per product so we don't overload the model.
Why
This is the bit that unlocks reading what's printed on the box. Without this, the model only sees the text the merchant wrote.
This run 1 image per product, sourced from the merchant's Shopify CDN.
5

Generate enhanced attributes with the AI

What
Send each product's text + image to gpt-5.4-nano in a single multimodal call. The prompts are written per client (see the wiki) so the output style stays on-brand.
Why
This is where the actual enhancement happens. For Metagenics, we generate four attributes: a rewritten title, a rewritten description, product highlights (bullet points), and product detail (structured spec triples).
This run 5 products × 4 attributes = 20 calls. Image-enabled. We also re-ran without images as a control — see comparison below.
6

Apply enhancement decisions

What
Merge the AI-generated values with the client's onboarding decisions — global defaults, per-record overrides, attribute mappings. This is the layer that ensures consistency across daily runs.
Why
The AI's output is one input among several. The tool needs to know, for example, "always set google_product_category to X for this client unless an override exists".
7

Write the supplemental feed

What
Output a CSV (or upload to a Google Sheet) containing the product id plus only the new and changed attributes. The merchant uploads this as a supplemental feed to Merchant Center, sitting on top of their primary feed.
Why
We never touch the merchant's primary feed — we add to it. They keep their existing pipeline intact; we layer the improvements on top.
This run See the linked Google Sheet at the top. Tab 2 is the deliverable.

Honest limitations

LimitationWhat it means in practice
Rewriting the title means giving up the standard title field Google's rules say you can't use both the standard title and the AI-rewritten structured_title for the same product. So during onboarding, the merchant has to decide: keep their existing titles, or let us rewrite them. Same for descriptions.
Length limits live in the prompt, not the code Google caps title at 150 chars and description at 5000. We tell the model these limits in the prompt and it generally obeys, but if it ever overshoots, the tool won't auto-truncate. The merchant review of the supplemental feed catches this before upload.
Scraping product pages adds time Adds roughly 3–10 seconds per product, longer for sites that need a real browser (e.g. heavy JavaScript). Not a problem for daily runs (we only re-scrape new or changed products) but it means onboarding 500 products takes some patience.
Image input only helps if the PDP actually has good packaging photos If the merchant's site only has lifestyle shots, or the packaging photo is low-resolution, the image input contributes less. Most quality DTC sites are fine; some legacy sites are not. The pipeline handles "no image found" cleanly — it just falls back to text-only.
Feed source quality still matters If the PDP itself is sparse and there's no decent packaging photo, image input can't rescue it. This is where the planned pre-onboarding audit step (per the kind of work James has been running) becomes the right answer — flag the structural data gaps that the merchant must fix at source before any AI enhancement gets involved.

How this would run in production

Three things worth knowing about the running model:

Cost

Running the AI step on a typical 300-product feed costs only a few dollars in OpenAI usage per full refresh. Adding image input increases that by roughly 30–50%. For daily refreshes, the per-day cost is far smaller, because we only generate attributes for products that are new or have changed — see "Memory" below.

Memory

The tool remembers what it generated for each product. A daily run compares today's feed to yesterday's, identifies new and changed products, and only re-generates for those. Everything else is reused from memory. This means a stable 300-product feed costs almost nothing day-to-day — costs only spike when the merchant adds new SKUs.

Automation

The whole pipeline — from reading the merchant's feed to producing the supplemental output — can run unattended on a schedule (e.g. nightly). We haven't put it on a schedule yet: that's a hosting decision once 2–3 clients are live and we know what the cadence should look like (daily vs. weekly, time of day, retry handling, etc.).

Delivery to Merchant Center

The supplemental feed can be delivered to Merchant Center via the standard supplemental feed mechanisms — CSV upload, scheduled fetch from a URL, or via a Google Sheet. The Google Sheet route (like the one linked at the top of this report) is particularly nice for clients who want to glance over what's about to go live before it's published.
